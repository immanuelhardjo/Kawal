## ADDED Requirements

### Requirement: Event record, owned

The system SHALL model an Event owned by exactly one user (`owner_user_id`), scoped to a case in the same user's dossier, with a date, a type (e.g., `hearing`, `indictment`, `verdict`, `asset_seizure`, `public_statement`), a title, a Bahasa Indonesia Markdown summary, a certainty label, at least one source id (referencing a Source owned by the same user), and a list of entity ids involved (entities owned by the same user).

#### Scenario: Event without sources rejected

- **WHEN** any code path attempts to persist an event with an empty source-id list
- **THEN** the system rejects the write

#### Scenario: Cross-user reference rejected

- **WHEN** a write attempts to persist an event whose source id or entity id refers to a row owned by a different user
- **THEN** the system rejects the write at validation

### Requirement: Garis Waktu rendered as a React Flow timeline

The system SHALL render the Garis Waktu (timeline) as a React Flow canvas with Event nodes laid out along a date-ordered track. Each event node SHALL display the date, the headline, the certainty chip, and at least one source link — with certainty and source visible before the summary text in reading order. Tapping an event node SHALL expand the node (inline or via the linked Profil panel) to show the 1-paragraph plain-Bahasa-Indonesia summary and the verbatim source excerpt.

#### Scenario: Rendering order

- **GIVEN** a user's case with events on 2024-03-01, 2024-05-12, and 2025-01-20
- **WHEN** the timeline is rendered for that case
- **THEN** the React Flow canvas lays out three event nodes along the date axis in chronological order
- **AND** the certainty chip and source link on each node visually precede the summary text

#### Scenario: Tap expand

- **WHEN** the user taps a timeline event node
- **THEN** the node expands inline (or surfaces in the Profil panel) showing the verbatim source excerpt, the source URL, and the archive URL, without navigating away from the Kasus Detail screen

### Requirement: Timeline drives the relationship-graph as-of-date

The system SHALL treat the user's current selection within the Garis Waktu — the focused event node, or the position of a draggable timeline cursor — as the single `asOfDate` for the Kasus Detail screen. The Peta Kasus relationship graph SHALL render only entities and relationships active at the timeline-driven `asOfDate`. There SHALL NOT be a separate scrubber widget for the graph.

#### Scenario: Tapping a timeline event sets the graph's date

- **WHEN** the user taps an event node dated 2024-06-15 in Garis Waktu
- **THEN** the Kasus Detail `asOfDate` becomes 2024-06-15
- **AND** the Peta Kasus re-renders to show only nodes and edges active on that date

#### Scenario: Dragging the timeline cursor updates the graph continuously (debounced)

- **WHEN** the user drags the timeline cursor across multiple dates
- **THEN** the `asOfDate` updates as the cursor moves, debounced according to the configured interval
- **AND** the Peta Kasus re-renders for each settled `asOfDate`

#### Scenario: No standalone graph scrubber

- **WHEN** the Kasus Detail screen is inspected
- **THEN** no UI control exists that scrubs the graph's date independently of the timeline

### Requirement: Timeline filters

The system SHALL support three filter axes on the timeline: by entity (one or many), by certainty label (any subset of the five labels), and by event type (any subset of the configured types). Filters SHALL compose as logical AND across axes and as logical OR within an axis. Selected filters SHALL also apply to which Event nodes are eligible to set the `asOfDate`.

#### Scenario: Combining filters

- **WHEN** the user filters by entity = "PT X" AND certainty in {`established`, `alleged`} AND event_type in {`hearing`}
- **THEN** the timeline shows only hearing events involving PT X whose certainty is established or alleged

#### Scenario: Linkage from graph tap

- **WHEN** the user taps a node in the Peta Kasus corresponding to entity E
- **THEN** the timeline filter is set to entity = E and the timeline re-renders

### Requirement: Branch View near case end

The system SHALL render a Branch View as part of the timeline section when the case is in `verdict` or `appeal` state, showing 2–4 plausible forward scenarios generated by the AI scenario port, each labeled clearly as a projection. The Branch View SHALL NOT render in earlier lifecycle states.

#### Scenario: Branch View visibility by state

- **GIVEN** a user's case currently in `trial`
- **WHEN** the timeline is rendered
- **THEN** no Branch View is shown

- **GIVEN** a user's case currently in `verdict`
- **WHEN** the timeline is rendered
- **THEN** the Branch View shows 2–4 scenarios, each prefixed with a "Proyeksi (bukan prediksi)" label, and each carrying a brief rationale and a list of next likely steps

#### Scenario: Scenarios are not persisted as facts

- **WHEN** the Branch View renders scenarios
- **THEN** none of the scenario text is written into the `claims`, `events`, or `relationships` tables (or their revision tables); scenarios are derived on read

### Requirement: Append-only revision history for Events

The system SHALL retain every create, update, and tombstone of an Event as a revision in an `events_revisions` table, with the standard revision-row shape (`revision_no`, `valid_from`, `valid_to`, actor `user_id`, `change_kind`, full payload).

#### Scenario: Owner browses event history

- **WHEN** the owner requests the revision history for one of their events
- **THEN** the response lists every revision in `revision_no` order with its payload, change_kind, valid_from, valid_to, and actor
